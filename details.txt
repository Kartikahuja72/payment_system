2. Core Principles

A. Payment Must Be Stateful

Never treat payments as simple API calls.

Each payment should have a lifecycle:

CREATED
PENDING
PROCESSING
AUTHORIZED
CAPTURED
FAILED
REFUNDED
REVERSED
SETTLED

Every transition must be persisted.

  "currency": "INR",

Step 2: Generate Idempotency Key

Critical.

User may:

click twice
refresh

Use:

Idempotency-Key: uuid

Store in Redis/MySQL:

key -> payment_id

return previous response
NEVER create duplicate payment

6. Transactional Outbox Pattern (VERY IMPORTANT)

Biggest issue:

DB updated
BUT Kafka publish failed

or:

Kafka published
BUT DB rollback happened

This causes money inconsistency.

Solution: Outbox Pattern

Inside SAME DB transaction:

BEGIN;

INSERT INTO payments(...);

INSERT INTO outbox_events(...);

COMMIT;

Then background worker publishes outbox events to Kafka.

This guarantees:

no lost events
atomicity


7. Kafka Consumer Design

Consumers must be:

idempotent
retry-safe
replay-safe


Consumer Rules
NEVER:
consume -> process -> commit
ALWAYS:
consume
persist result
commit offset
Deduplication

Every Kafka event contains:

{
  "event_id": "uuid",
  "payment_id": "123"
}

Consumer stores processed event IDs.

If replay happens:

 ->ignore duplicate

9. Gateway Failure Handling

Payment gateways fail frequently.

timeout

10. MOST IMPORTANT EDGE CASE
Gateway Timeout After Debit

Scenario:

User paid
Bank debited money
Gateway timed out

Your system DOES NOT know success/failure.

NEVER mark failed immediately.

Mark:

UNKNOWN / PENDING_VERIFICATION

Then:

poll gateway
wait webhook
reconciliation later

This is where most payment bugs happen.


11. Webhook System

Gateways send async webhook.

Must support:

duplicate webhook
delayed webhook
out-of-order webhook
Webhook Design

Webhook service:

validates signature

ACKs quickly
pushes to Kafka

Never process heavy logic synchronously.


13. Double Entry Ledger (CRITICAL)

Never trust payment table for balance.

Use accounting ledger.

Ledger Entries
User Wallet      -100
Platform Holding +100

Every transaction:

debit + credit
sum = 0

Immutable ledger.

Never UPDATE balances directly.


14. Exactly Once Processing

True exactly-once doesn't fully exist distributedly.

Use:

at-least-once Kafka
idempotent consumers
dedupe keys

This gives EFFECTIVE exactly-once.

If event fails repeatedly:

17. Reconciliation Engine (SUPER IMPORTANT)

Banks and gateways WILL mismatch.

Daily reconciliation job compares:

Internal DB
vs
Gateway settlement reports
vs
Bank reports

Cases:

money deducted but no record
success internally but failed at gateway
refund mismatch

This is what actually saves money in production.
